TITLE: Plaintext as the primary format for project text files

DATE: 2026-04-09

STATUS: accepted

AUTHOR: Andrii Chemerysov


CONTEXT

The project includes several categories of human-authored text files: project
documentation under docs/, root-level files such as README and LICENSE, and any
similar files added as the project grows. A format decision made implicitly and
inconsistently across these files creates unnecessary fragmentation. The format
should be chosen explicitly and applied uniformly.

DECISION

Human-authored text files across the project, including docs/, README, LICENSE,
and equivalent files, are authored as plaintext (.txt). Structure is enforced by
convention and whitespace rather than markup syntax. No rendering step is
required to read any file in its intended form.

ALTERNATIVES CONSIDERED

Markdown (.md): Rejected as the primary format. Markdown renders well on
platforms like GitHub but carries visible syntax noise in raw form. For prose
documents like ADRs, READMEs, and license files, where structure is already
enforced by convention, Markdown's additions are cosmetic and
rendering-dependent. Markdown remains appropriate for files that will be
rendered as part of a documentation site, if the project acquires one, and may
be revisited at that point.

reStructuredText (.rst): Rejected. More expressive than Markdown but carries a
steeper learning curve. The overhead is not justified by the project's current
scope or tooling.

AsciiDoc (.adoc): Rejected. Powerful and well-specified but meaningfully more
complex syntax than Markdown. Appropriate for large documentation projects. Not
appropriate here.

TOML or YAML frontmatter with plaintext body: Rejected for now. The hybrid
approach, machine-readable structured metadata at the top with freeform prose
below, enables tooling to query document metadata without parsing prose. This is
appealing as the docs/ folder grows. Rejected because it introduces a parsing
dependency and a format split that adds friction without sufficient payoff at
the project's current scale. A transition remains a possibility if tooling needs
emerge.

RATIONALE

Plaintext is the only format that requires no tooling, no rendering step, and no
ecosystem knowledge to read in its intended form. It is readable in any
terminal, any editor, and any file browser. It is durable across decades without
format drift. Applying it uniformly across all human-authored text files
eliminates format fragmentation and makes the rule simple to follow: plaintext
by default, exceptions recorded as decisions.

CONSEQUENCES

Positive: all text files are readable anywhere without tooling or rendering. The
format imposes no learning curve and no ecosystem dependency. Files remain
consistent and legible across the full history of the repository.

Negative: plaintext offers no native support for inline code formatting, links,
or tables. Documents requiring those elements will either work around the
limitation with convention or require a format exception to be recorded as a
separate decision.

Neutral: if the project acquires a documentation site or tooling that heavily
benefits from Markdown or structured frontmatter, this decision will be
reconsidered.
