TITLE: Build-time generation as the strategy for server-side frontend HTML

DATE: 2026-04-18

STATUS: accepted

AUTHOR: Andrii Chemerysov


CONTEXT

Prior decisions establish a hybrid rendering model
(2026-04-16-hybrid-as-rendering-model) in which static content regions of model
pages are pre-rendered to HTML and interactive regions are rendered client-side
as React components (2026-04-18-react-as-ui-component-library). The hybrid
rendering decision defers the question of when and how the pre-rendered HTML is
produced.

DECISION

Frontend HTML is generated at build time via static generation. No JavaScript
runtime runs in production for the purpose of generating HTML at request time.
The Go application server serves the statically generated HTML files and
JavaScript bundles produced by the frontend framework's build step. The build
pipeline is triggered by changes to written content, not by data events. Data
freshness of interactive elements is maintained continuously by the Go server's
API independently of the build pipeline.

ALTERNATIVES CONSIDERED

Per-request generation: HTML is produced by executing the React component tree
against current data on every incoming page request. Pages always reflect the
latest available data without a build pipeline trigger. Per-user personalization
can appear in the initial HTML without a client-side fetch. Rejected because the
capabilities this approach provides over build-time generation are not required
by any page the project contains. Research and methodology pages update when
data events occur, not when users arrive, and a build triggered by a content
change produces HTML that is correct for every subsequent reader until the next
change. Live updating pages deliver their dynamic content via the Go server
after the static shell loads, meaning the shell itself has no per-request
generation requirement. The analytical playground reconstructs user
configuration client-side from URL parameters or the Go API, meaning its initial
shell is the same for every user and requires no per-request computation. The
operational cost of a permanently running process capable of executing
JavaScript, its memory consumption, supervision requirements, and expanded
failure surface, is not justified by capabilities the project does not require.

Incremental static regeneration: A middle ground in which pages are statically
generated at build time but individual pages can be regenerated in the
background on a schedule or on demand, without triggering a full rebuild of the
entire site. A stale page is served immediately while the fresh version is
generated behind it, and subsequent requests receive the fresh version. Rejected
because it introduces a hybrid of build-time and request-time generation that
adds operational complexity without solving any problem the project has. Written
content changes are rare and a full rebuild on those occasions is not
burdensome. Data freshness is handled by the Go API independently of the build.
There is no latency problem for incremental regeneration to solve.

Edge-side generation: HTML is generated at geographically distributed edge nodes
close to users rather than on the origin server, reducing the network component
of response latency. Rejected as a primary generation strategy because the
Python engine and Go server run in one location, and pages requiring fresh data
from those systems still incur a round trip to that location regardless of where
the HTML is generated. For content that does not require per-request data
fetching, which is precisely the content build-time generation handles
correctly, edge distribution of the statically generated files via a CDN is a
compatible and additive optimization that does not conflict with this decision
and does not require revisiting it.

RATIONALE

Build-time generation is the correct strategy when the content of a page is
determined by data that exists before any user requests it. The written content
of model pages exists when contributors author it. The interactive elements
source their data from the Go API at runtime and are not part of the HTML that
the build produces. In both cases the HTML can be produced before users arrive
and served as a file. The decision is consistent with the reasoning in the
2026-04-11-not-python-as-backend-language ADR, which rejected a runtime-heavy
process for the application server on memory and operational grounds. It is
consistent with the choice of Go as the application server in
2026-04-11-go-as-backend-language, whose value proposition includes a low memory
footprint that leaves headroom for the calculation engine. A permanently running
JavaScript process in production would partially undermine both of those
decisions.

CONSEQUENCES

Positive: no JavaScript runtime runs in production. Memory pressure does not
increase beyond what those three processes require. The Go server serves HTML
files with negligible overhead. The operational surface does not grow. Build
failures are visible during deployment rather than manifesting as request-time
failures in production.

Negative: written content changes require a rebuild and redeployment before
readers see them. For corrections or rewrites this latency is acceptable, but
the pipeline must be reliable and fast enough that contributors are not blocked.
Per-user personalization cannot appear in the initial HTML without a client-side
fetch, meaning any personalized elements are rendered after the static shell
loads.

Neutral: this decision does not preclude introducing a JavaScript runtime in
production at a later stage if requirements emerge that require it. The
framework decision that follows must be compatible with static output mode,
meaning it must be capable of producing a folder of HTML files and JavaScript
bundles with no server process required at runtime.
