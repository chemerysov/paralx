TITLE: Editor tooling policy and committed editor configuration

DATE: 2026-04-09

STATUS: accepted

AUTHOR: Andrii Chemerysov


CONTEXT

The project requires a stance on editor tooling. The repository contains a
.vscode/ folder reflecting the editor in use during initial setup. Without an
explicit policy, the basis on which other editors' configuration would be
welcome is unclear to future contributors.


DECISION

The project does not designate a preferred or recommended editor. Contributors
use whatever editor they choose. Editor-specific configuration files and folders
may be committed when they encode project-level conventions useful to other
contributors using the same editor. The .vscode/ folder is committed with
personal machine state excluded via .gitignore. Contributors using other editors
are welcome to commit equivalent configuration under the same bar: project-level
usefulness, not personal preference. It is the contributing contributor's
responsibility to ensure the corresponding .gitignore rules exclude personal
state files for their editor.


ALTERNATIVES CONSIDERED

Omit .vscode/ entirely: Would enforce strict editor neutrality at the cost of
discarding configuration that provides concrete value to contributors using
VS Code. Rejected because the folder's presence imposes no cost on contributors
using other editors.

Commit .vscode/ without a policy: Leaves the principle behind the folder's
presence unrecorded, making it unclear whether other editors' configuration
would be welcome and on what basis. Rejected for that reason.


RATIONALE

Editor choice is a contributor-level decision the project has no basis for
constraining. The .vscode/ folder contains configuration useful to contributors
using VS Code and its presence imposes no cost on anyone else. The distinction
between project-level configuration and personal machine state provides a clear
and applicable bar for what should and should not be committed, with .gitignore
as the enforcement mechanism. Placing the responsibility for .gitignore hygiene
on the contributor adding the configuration keeps that burden local to whoever
is best positioned to know what their editor's files mean.


CONSEQUENCES

Positive: contributors using VS Code have committed workspace configuration
immediately available, scoped to project-level files. The project's editor
neutrality is explicit. Contributors using other editors have a clear and
non-arbitrary basis for contributing equivalent configuration.

Negative: the responsibility for correct .gitignore scoping rests with
individual contributors and is not centrally enforced. A contributor unfamiliar
with their editor's file layout may commit personal state inadvertently. If
editor configuration folders accumulate and begin to clutter the repository
root, excluding all of them via .gitignore would be an appropriate response.

Neutral: as the project grows, the .gitignore may require refinement if VS Code
adds new file types within .vscode/ whose status is ambiguous. That would be
maintenance rather than a new decision.
